# テストフレームワークの決定

## 目的

Brodo で使用する C++ テストフレームワークを決める。

## 現在の状態

decided

## 議論の前提

この議論は、次の前提を置いて始める:

- Brodo は C++20 と CMake を前提に進める想定である。
- 最初のテストは、描画や OS 依存の統合テストよりも、エンジン基盤のユニットテストを優先する。
- Windows + MSVC が最初の開発環境であり、CI も当面はその前提で考える。
- まだテスト基盤は存在しないため、導入コストの低さも評価対象に含める。

## 決定

Brodo の C++ テストフレームワークには GoogleTest を採用する。

理由:

- C++ プロジェクトで広く使われている。
- CMake と組み合わせやすい。
- エンジン基盤のユニットテストに向いている。

## 候補

### GoogleTest

採用する。

トレードオフ:

- エコシステムとドキュメントが強い。
- 多くの C++ 開発者に馴染みがある。
- 小さな single-header 系フレームワークよりは少し重い。

### Catch2

より軽く、表現力のあるテスト記述を重視する場合の有力な代替候補。

トレードオフ:

- 読み書きしやすい。
- プロジェクト設定が比較的小さい。
- 大規模な C++ エンジンコードベースでは GoogleTest ほど標準的ではない。

## 判断基準

この決定では、少なくとも次の観点で比較する:

- CMake への統合が単純か。
- Windows + MSVC で素直に導入できるか。
- テスト発見、実行、失敗時の出力が扱いやすいか。
- math や utility のような小さなユニットテストを素早く増やせるか。
- 将来 CI に載せるときに追加コストが小さいか。
- 新しく参加する C++ 開発者にとって馴染みがあるか。

## 決定理由

GoogleTest を採用する判断は妥当である。

理由:

- CMake と組み合わせた導入例が多く、初期セットアップで迷いにくい。
- 将来 CI を追加するときの構成を想像しやすい。
- エンジン開発で増えやすい素朴なユニットテストとの相性がよい。

ただし、次の条件を強く重視する場合は、将来 Catch2 を再評価してよい:

- 依存をできるだけ軽くしたい。
- 単一バイナリまたは少ない設定で素早く始めたい。
- テスト記述の簡潔さを標準性より優先したい。

## まだ残っている質問

1. 最初から CMake 経由でテストを統合するか？
2. 最初のテスト対象は math、memory、asset loading、platform code のどれにするか？

## 次の一歩

GoogleTest を採用した前提で、次の順で進める:

1. 最初のテスト対象を `math` か `utility` に絞る。
2. GoogleTest を前提に最小構成の CMake 統合案を書く。
3. その案が過剰に感じる場合だけ Catch2 案と比較する。
